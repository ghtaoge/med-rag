# Med-Rag 输入安全、身份权限与管理操作保护设计

## 1. 背景

Med-Rag 当前提供医疗知识问答、文档上传、同步、删除和索引能力，但尚未形成完整的身份、授权和数据防泄露边界。用户问题会直接进入意图识别、检索和 LLM；文档管理接口缺少角色授权；上传文件名直接参与本地路径拼接；普通日志也没有统一的敏感信息处理策略。

本设计是 Med-Rag 功能演进计划的第一个子项目。后续两个子项目依次为：

1. 文档解析质量评测与可插拔解析服务。
2. 逐句证据链、来源冲突检测与安全拒答。

安全底座必须先完成，确保后续解析结果和可信回答始终服从相同的部门权限、文件隔离和审计规则。

## 2. 目标

- 为问答和全部管理接口增加身份认证。
- 按部门和固定角色控制知识文档、会话及管理操作。
- 在检索发生前排除未授权文档，防止内容进入 Prompt、回答、引用或日志。
- 对用户输入进行规范化、敏感信息检测、攻击意图识别和分级处置。
- 对生成结果进行敏感信息与越权内容扫描。
- 隔离并校验上传文件，防止路径穿越、伪造格式、恶意文件和资源耗尽。
- 为高风险操作和安全事件提供可追溯、不可静默绕过的审计记录。
- 保持身份接口与 OIDC 兼容，为未来接入企业单点登录预留边界。

## 3. 非目标

- 本阶段不接入患者病历，不处理患者级临床决策。
- 本阶段不允许问答链路调用外部业务系统或执行平台管理操作。
- 本阶段不实现用户自定义角色或通用 ABAC 策略语言。
- 本阶段不选定新的文档解析主引擎；只定义解析服务必须遵守的安全边界。
- 本阶段不实现逐句证据验证；可信回答由后续独立规格负责。

## 4. 总体架构

安全控制分为六层。除健康检查、登录、刷新 Token 和注销外，所有 `/api/v1/*` 接口都要求有效 Access Token；刷新和注销通过受保护的 Refresh Token Cookie 完成：

```text
客户端
  -> 身份认证
  -> API 授权
  -> 输入 SafetyGateway
  -> 带权限过滤的检索和生成
  -> 输出 DLP
  -> 审计与监控
```

文档进入知识库的链路独立于问答链路：

```text
隔离区上传
  -> 文件签名与 MIME 校验
  -> 恶意内容与资源限制检查
  -> 沙箱解析
  -> 解析质量检查
  -> 人工审核
  -> 发布并建立带权限元数据的索引
```

任何层失败都采用默认拒绝策略。系统不得通过关闭校验、切换到权限更宽的解析器或使用 LLM 通用知识来静默降级。

## 5. 身份认证

### 5.1 本地账户

第一阶段提供本地账户，但认证服务通过抽象接口向业务层暴露统一的用户身份和声明，后续可替换为 OIDC Provider。

- 密码使用 Argon2id 哈希，并为每个密码使用独立随机盐。
- Access Token 有效期为 15 分钟。
- Refresh Token 有效期为 7 天，服务端只保存其哈希，并在每次刷新时轮换。
- 注销、密码修改、账户禁用和疑似 Token 重放都会撤销相关 Refresh Token 链。
- 连续 5 次登录失败后账户锁定 15 分钟；登录接口同时接受 IP 级限速。
- 首次登录和管理员重置后必须修改密码。
- 高风险管理操作要求最近 5 分钟内完成过密码再认证。

Access Token 通过 `Authorization: Bearer` 发送。Refresh Token 存放在 `HttpOnly`、`Secure`、`SameSite=Strict` Cookie 中。所有使用 Cookie 改变状态的接口都校验 CSRF Token。

### 5.2 持久化

生产环境使用 PostgreSQL 保存账户、部门、角色、文档元数据、Token 撤销状态和审计记录。本地开发和单元测试允许使用 SQLite。Redis 仅用于限流、缓存和短期会话，不作为权限或审计的唯一数据源。

## 6. 授权模型

### 6.1 部门和角色

用户可以属于一个或多个部门，角色通过部门成员关系授予。固定角色如下：

| 角色 | 权限范围 |
|---|---|
| `reader` | 在授权部门范围内问答、查看自己的历史会话和可见文档元数据 |
| `knowledge_editor` | 上传草稿、编辑本部门文档元数据、发起审核，不能发布自己的修改 |
| `knowledge_reviewer` | 审核、发布和撤回本部门文档，发布人必须不同于最后编辑人 |
| `department_admin` | 管理本部门成员及角色，不能读取未授权部门文档 |
| `security_auditor` | 只读查看安全事件和审计记录，不获得文档内容读取权限 |
| `platform_admin` | 管理全局运行配置，不得绕过部门可见范围和文档审核状态 |

权限检查以服务端数据库中的有效成员关系为准，绝不信任客户端传入的角色、部门或文档范围。

### 6.2 文档可见范围

每个文档版本至少包含：

- `owner_department_id`
- `visible_department_ids`
- `visibility`: `department_only` 或 `shared_departments`
- `review_status`: `draft`、`approved`、`expired` 或 `revoked`
- `created_by`、`last_edited_by`、`reviewed_by`
- `version`、`file_hash`、`published_at`、`expires_at`

只有 `approved`、未过期且用户部门可见的版本能够参与线上检索。撤回和过期文档从活动索引中移除，但保留原文件、版本元数据和历史审计引用。

### 6.3 检索强制过滤

授权范围必须在多路召回前生效：

- Milvus 查询使用文档 ID、部门和审核状态过滤表达式。
- Whoosh 索引增加相同的权限字段，并在关键词查询中应用过滤。
- RRF 融合前验证每一路结果均属于允许范围。
- 融合后、进入 Prompt 前再次执行防御性授权断言。
- 检索缓存键包含用户的授权范围版本；角色变化后立即使相关缓存失效。

未经授权的内容不得以“结果数量”“文档名称”“摘要”“相似度”或错误信息的形式泄露。

## 7. 输入 SafetyGateway

### 7.1 处理阶段

所有聊天入口在意图识别前执行以下步骤：

1. Unicode 规范化，移除不允许的控制字符和不可见字符。
2. 校验字符集、长度、重复率和异常混淆字符。
3. 检测密码、Token、API Key、身份证号、手机号、邮箱和高熵密钥。
4. 检测系统提示套取、忽略权限、批量资料枚举、内部数据导出、路径或命令注入，以及诱导执行管理操作。
5. 结合规则、统计特征和本地语义分类器生成风险结果。

危险词只作为风险特征，不能单独触发阻断。医学问题可能合法包含“毒性”“致死”“禁忌”等词，因此风险判断必须考虑意图、请求动作和数据范围。

### 7.2 风险结果

`SafetyAssessment` 包含：

- `risk_level`: `low`、`medium` 或 `high`
- `categories`: 命中的风险类别列表
- `matched_signals`: 可审计但不向普通用户暴露的信号
- `redacted_input`: 脱敏后的输入
- `policy_version`
- `decision`: `allow`、`allow_restricted` 或 `block`

处置规则：

- `low`: 在用户既有授权范围内正常检索。
- `medium`: 使用脱敏输入；禁止批量列举和大段原文输出；最多返回 3 份文档，每份原文预览不超过 300 个字符；强制执行输出 DLP。
- `high`: 不进入检索或模型，返回稳定的安全错误码并写入安全事件。
- 分类器不可用或超时时：进入保守模式，只允许不超过 500 个字符的普通知识问答；疑似敏感、枚举或管理请求被阻断。

SafetyGateway 不能扩大用户原有权限。任何“限制检索”都只会进一步收紧范围。

## 8. 输出防泄露

答案、引用和错误响应在发送前经过输出 DLP：

- 检测并遮蔽密码、Token、API Key 和个人敏感信息。
- 校验每个引用的文档 ID 和版本仍属于当前用户的授权范围。
- 检测大段原文复制、跨文档批量枚举和内部路径泄露。
- 阻断系统提示、模型配置、数据库连接信息和内部堆栈。
- SSE 流式输出使用安全缓冲窗口，不直接发送未经扫描的完整敏感片段。缓冲区不得小于 512 个 Unicode 字符，并且不得小于已启用检测器支持的最长模式长度。

若流式过程中发现高风险内容，服务端发送 `safety_blocked` 事件并终止生成。已输出内容不得包含完整敏感值；安全缓冲窗口必须足以覆盖受支持的密钥和标识符模式。

## 9. 管理操作保护

问答编排器不持有上传、删除、同步、审核、权限修改或配置修改能力。管理操作只能通过独立 API 发起，并同时满足：

- 有效身份和对应部门角色。
- CSRF 校验和最近 5 分钟内的再认证。
- 操作原因和幂等键。
- 对发布、撤回、权限变更和配置修改进行明确确认。
- 发布人不同于最后编辑人。
- 产生成功或失败审计记录。

已发布文档不能物理删除。撤回操作会生成新状态、移出活动索引并保留历史版本。只有受控的数据保留任务能够在满足保留期后执行物理清理。

## 10. 文件上传与解析隔离

### 10.1 安全存储

- 服务端生成 UUID 文档 ID 和存储键。
- 原始文件名经过 Unicode 规范化和显示字符过滤后，仅作为元数据保存。
- 上传、同步、下载和撤回接口全部使用文档 ID，不使用用户文件名定位路径。
- 隔离区、原始文件区、解析结果区和已发布索引区使用不同目录或对象存储前缀。

### 10.2 入库校验

文件发布前必须完成：

- 真实文件签名、声明 MIME、扩展名和内部结构一致性检查。
- 文件大小、页数、解压后大小、压缩比、图片像素、工作表行列数和处理时间限制。
- Office 宏、外部关系、嵌入对象和异常压缩成员检查。
- PDF JavaScript、启动动作、附件和异常对象检查。
- 恶意软件扫描；扫描失败或服务不可用时保持隔离。

具体限制值作为部署配置提供，但不得高于系统硬上限。硬上限为：单文件 50 MB、PDF 1000 页、解压后 500 MB、压缩比 100:1、单张图片 10000 x 10000 像素、单个工作表 200000 行乘 500 列、单文件解析 10 分钟。

### 10.3 解析 Worker

解析 Worker 独立于主 API 部署，并满足：

- 非特权用户、只读根文件系统和独立临时目录。
- 默认禁用网络，只允许访问任务所需的本地模型和对象存储代理。
- CPU、内存、GPU、磁盘和处理时间配额。
- 每个任务结束后清理临时文件。
- 不接受远程 URL，不自动加载第三方插件，不执行文档中的宏或脚本。
- 失败时返回稳定错误码，不返回内部路径、命令行或堆栈。

MarkItDown 仅允许使用受控字节流或隔离区本地文件，禁用插件和远程 URI。Pandoc 只能作为沙箱中的格式转换子进程。Docling、PaddleOCR、MinerU 或其他解析器均通过相同的 Worker 接口和资源策略调用。

## 11. 数据模型

关系数据库至少包含以下实体：

- `users`
- `departments`
- `department_memberships`
- `roles`
- `refresh_token_families`
- `knowledge_documents`
- `knowledge_document_versions`
- `document_visible_departments`
- `review_actions`
- `safety_events`
- `audit_events`

审计事件记录操作者、部门、动作、资源 ID、结果、原因、请求 ID、策略版本、时间和前后状态摘要。安全事件保存输入指纹和脱敏摘要，不在普通日志中保存完整敏感输入。

## 12. API 变化

新增认证接口：

- `POST /api/v1/auth/login`
- `POST /api/v1/auth/refresh`
- `POST /api/v1/auth/logout`
- `POST /api/v1/auth/change-password`
- `POST /api/v1/auth/reauthenticate`
- `GET /api/v1/auth/me`

文档接口改为使用文档 ID：

- `POST /api/v1/documents`
- `GET /api/v1/documents/{document_id}`
- `POST /api/v1/documents/{document_id}/submit-review`
- `POST /api/v1/documents/{document_id}/approve`
- `POST /api/v1/documents/{document_id}/revoke`
- `POST /api/v1/documents/{document_id}/sync`

旧文件名接口在迁移期只接受管理员调用，并返回弃用响应头；一个发布周期后移除。旧接口不得继续直接拼接本地路径。

聊天接口在不破坏现有成功响应的前提下增加 `safety` 摘要和 `request_id`。被阻断请求返回 `403` 与稳定业务码 `SAFETY_POLICY_BLOCKED`；未认证返回 `401`；无资源权限统一返回 `404`，避免泄露资源是否存在。

## 13. 日志、审计与隐私

- 普通请求日志只记录请求 ID、用户 ID、部门、路径、状态码和耗时。
- 不记录密码、Token、Cookie、Authorization Header、完整问句、完整答案或原始文档内容。
- 安全事件记录风险类别、决策、策略版本、输入哈希和脱敏摘要。
- 权限变化、文档发布、撤回、同步、登录、Token 撤销和安全阻断均写入审计表。
- 必须审计的管理操作、安全阻断或权限变更在审计写入失败时一律失败关闭。普通低风险问答不生成安全审计事件，只记录不含正文的运行指标。
- 审计查询只对 `security_auditor` 开放，并记录审计查询本身。

## 14. 错误处理

- 客户端只接收稳定错误码和可操作提示，不接收内部异常文本。
- 身份错误不区分“用户名不存在”和“密码错误”。
- 未授权资源统一表现为不存在。
- 检索权限过滤失败、输出 DLP 不可用或文档状态无法确认时终止回答。
- 文件扫描、解析或审核失败时文件保持隔离，不进入活动索引。
- PostgreSQL 不可用时，所有受保护接口统一返回 `503 AUTHORIZATION_SERVICE_UNAVAILABLE`。系统不得依赖可能过期的本地权限缓存继续问答或管理操作。

## 15. 测试策略

### 15.1 自动化测试

- 单元测试：密码、Token 轮换、角色矩阵、风险决策、脱敏和文档状态转换。
- 属性与模糊测试：文件名、路径、Unicode、超长输入、压缩包成员和异常文件结构。
- 集成测试：PostgreSQL、Redis、Milvus 和 Whoosh 之间的权限一致性。
- 端到端测试：上传隔离、审核发布、受限检索、撤回和历史审计。
- 对抗测试：Prompt Injection、系统提示套取、跨部门枚举、批量导出和管理操作诱导。
- 日志测试：断言敏感值不会出现在应用日志、异常响应或 SSE 事件中。

### 15.2 验收指标

- 未授权文档进入检索结果、Prompt、答案或引用的次数为 0。
- 路径穿越、伪造 MIME、压缩炸弹和超限文件测试全部阻断。
- 预置密钥、Token 和个人敏感信息的输出拦截率为 100%。
- 高风险数据套取与权限绕过攻击拦截率不低于 95%。
- 正常医疗问题误拦截率不高于 2%。
- 管理操作审计完整率为 100%。
- 普通应用日志中的原始密码、Token、密钥和完整敏感问句数量为 0。

任一数据隔离、密钥泄露、路径安全或审计完整性指标不达标时不得发布。

## 16. 分阶段上线

### 阶段 0：紧急安全修复

- 修复文件名路径越界。
- 增加文件签名、MIME 和资源上限校验。
- 清理敏感日志和内部异常输出。
- 临时保护上传、删除、同步和评估管理接口。

### 阶段 1：身份与部门权限

- 引入 PostgreSQL 和认证服务。
- 实现部门成员关系、固定角色和文档可见范围。
- 将权限过滤下沉到 Milvus 和 Whoosh。
- 迁移文档接口到 UUID。

### 阶段 2：安全网关与审计

- 实现输入规范化、DLP、攻击意图分类和分级策略。
- 实现输出 DLP、SSE 安全缓冲和统一安全事件。
- 上线安全审计查询和监控指标。

### 阶段 3：解析隔离

- 将文档解析迁移到隔离 Worker。
- 接入恶意文件扫描、资源配额和任务状态管理。
- 为下一阶段的解析器基准评测提供统一、安全的插件接口。

## 17. 后续子项目接口

文档解析质量子项目必须复用本设计的隔离 Worker、文档 ID、版本状态和部门权限字段。逐句证据链子项目必须复用授权检索、输出 DLP、文档版本和审计 ID，不得建立旁路检索或独立权限模型。
